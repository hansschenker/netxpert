# Cloudflare deployment

netxpert deploys to **Cloudflare Workers** using the
[`@cloudflare/vite-plugin`](https://developers.cloudflare.com/workers/vite-plugin/).

## The model

- Each deployable app (e.g. `apps/playground`) has its own **`wrangler.jsonc`**
  and a `vite.config.ts` that uses the Cloudflare plugin.
- In dev, `vite dev` runs your Worker in the **real Workers runtime**
  (workerd via Miniflare) — bindings and all — instead of a Node shim.
- `wrangler.jsonc` is the **input** config. At build time the plugin emits a
  separate **output** config that references the built assets.
- `vite build` produces both the Worker bundle and the client assets; deploy
  with `wrangler deploy`.

## The playground

`apps/playground` is the reference vertical slice. Its `vite.config.ts` is just:

```ts
import { cloudflare } from '@cloudflare/vite-plugin'
import { defineConfig } from 'vite'

export default defineConfig({
  plugins: [cloudflare()],
})
```

Its `wrangler.jsonc` wires the Worker entry and a static-assets binding:

```jsonc
{
  "name": "netxpert-playground",
  "main": "src/worker/index.ts",
  "compatibility_date": "2026-06-01",
  "compatibility_flags": ["nodejs_compat"],
  "assets": {
    "binding": "ASSETS",
    "not_found_handling": "single-page-application"
  },
  "observability": { "enabled": true }
}
```

The Worker serves API routes and falls back to static assets for everything else:

```ts
export default {
  async fetch(request, env): Promise<Response> {
    const url = new URL(request.url)
    if (url.pathname.startsWith('/api/')) {
      return Response.json({ framework: 'rxjs-full-claude', /* … */ })
    }
    return env.ASSETS.fetch(request)
  },
} satisfies ExportedHandler<Env>
```

## Worker types

Runtime types (`Env`, `Fetcher`, `ExportedHandler`, …) are generated by Wrangler,
not hand-written:

```bash
pnpm --filter @netxpert/playground cf-typegen   # wrangler types
```

This writes `worker-configuration.d.ts` (git-ignored). Re-run it after editing
`wrangler.jsonc`.

## Deploying

```bash
# build first, then deploy
pnpm build --filter @netxpert/playground
pnpm --filter @netxpert/playground deploy        # wrangler deploy
```

You'll need to authenticate Wrangler once (`wrangler login`) or provide a
`CLOUDFLARE_API_TOKEN` in CI.

## The docs site

This documentation site (`packages/docs/`) ships as an **assets-only Worker** —
no server code, just the static VitePress build:

```jsonc
// packages/docs/wrangler.jsonc
{
  "name": "netxpert-docs",
  "compatibility_date": "2026-06-01",
  "assets": { "directory": "./dist" }
}
```

```bash
pnpm --filter @netxpert/docs build
pnpm --filter @netxpert/docs deploy
```

## Multi-Worker full-stack (future)

When an app needs a separate API Worker, the Cloudflare plugin's
`auxiliaryWorkers` + service bindings let one `vite dev` run several Workers with
typed RPC between them. The `@rxjs-full/cloudflare` adapter is where that wiring
will live.
